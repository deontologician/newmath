export function fromSolverResult(op: number[][]): OpTable {
  const size = op.length;
  const cells = op.map(row => row.map(v => v - 1));  // Subtract 1 from each value
  return { size, cells };
}

## Implementation Plan: Constraint Selection UI

### Overview

Wire the existing MiniZinc constraint solver into the UI by adding a row of toggle buttons below the main controls. When constraints are selected, "New Math" generates a conforming table via the WASM solver. When none are selected, it falls back to the existing `randomTable()`. The design must handle the async solver (loading state), unsatisfiable configurations (error feedback), and keep the minimal dark aesthetic intact.

### Architecture Decisions

**Placement: Inline toggle row below controls.** A collapsible panel adds unnecessary interaction cost for only 6 options. Checkboxes would look out of place in the current design. Instead, use a row of small pill-shaped toggle buttons with the constraint's short mathematical notation. They sit below the existing button bar, visually subordinate but always visible. This matches the app's minimalist approach: everything on one screen, no hidden drawers.

**No constraints = random table (skip solver).** The `SolverConfig` Zod schema requires `.min(1)` on constraints. Rather than relaxing this validation, the UI simply calls `randomTable()` directly when the set is empty, which is also faster since it avoids WASM initialization.

**Loading state on the "New Math" button itself.** Rather than a separate spinner or overlay, the button text changes to "Solving..." and becomes disabled during solver execution. This keeps the UI clean and communicates state without additional visual elements.

**Unsatisfiable: brief inline message.** If the solver throws (unsatisfiable or error), display a transient message below the controls that fades out after a few seconds, e.g. "No table satisfies these constraints." No modal or alert.

Three commits:

newTableBtn.addEventListener('click', async () => {
  if (state.animating || state.solving) return;

  const size = 3;
  clearMessage();

  if (state.selectedConstraints.size === 0) {
    // No constraints: random table (fast path, no solver needed)
    randomizeGlyphs(size);
    state.table = randomTable(size);
    state.expr = randomExpr(size, 5);
    ensureReducible();
    renderAll();
    return;
  }

  // Solver path: show loading state
  state.solving = true;
  const originalText = newTableBtn.textContent;
  newTableBtn.textContent = 'Solving...';
  newTableBtn.classList.add('solving');

  try {
    const result = await solve({
      symbolCount: size,
      constraints: [...state.selectedConstraints],
    });

    randomizeGlyphs(size);
    state.table = fromSolverResult(result.op);
    state.expr = randomExpr(size, 5);
    ensureReducible();
    renderAll();
  } catch (err) {
    const msg = err instanceof Error ? err.message : 'Solver error';
    if (msg.includes('No satisfying')) {
      showMessage('No table satisfies these properties.');
    } else {
      showMessage('Solver error -- try different properties.');
      console.error('Solver error:', err);
    }
  } finally {
    state.solving = false;
    newTableBtn.textContent = originalText;
    newTableBtn.classList.remove('solving');

- **`state.solving` guard** prevents double-clicks during solve. Both `animating` and `solving` are checked.

- **try/catch/finally** ensures the button always returns to its normal state, even on error.
- **`fromSolverResult`** handles the 1-indexed to 0-indexed conversion.
- **Error message distinguishes unsatisfiable from other errors.** The `solve()` function in `src/minizinc/solve.ts` throws `"No satisfying operation table exists for this configuration"` for UNSATISFIABLE, which we detect via string match.
- **`ensureReducible()`** is called after table generation to make sure the expression has at least one redex, same as initialization.

### Step 4: Edge Cases and Error Handling

**Unsatisfiable combinations.** Some constraint combinations have no solution for small symbol counts. For example, with 2 symbols, requesting all 6 constraints simultaneously may be unsatisfiable. The solver correctly throws and the UI shows a message. The existing table remains unchanged so the user can continue playing with it.

**First-time solver initialization.** The `ensureInit()` in `solve.ts` is idempotent and lazy. The first "New Math" click with constraints will incur a one-time WASM load (~1-2 seconds). The "Solving..." button state covers this. Subsequent solves are faster. No preloading is needed since the user might never select any constraints.

**Rapid clicking.** The `state.solving` guard prevents re-entry. The `state.animating` guard (already existing) prevents clicks during reduction animations.

**Browser back/forward.** The app is a single-page app with no routing. Constraint selection is ephemeral state in memory, which matches the current approach for table and expression state.
