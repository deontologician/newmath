/**
 * Render a list of leaf symbol values as a flat expression with no parens.
 * Every operator between adjacent symbols is a clickable redex target.
 */
export function renderExprFlat(leaves: number[]): FlatRenderResult {
  const opIndexMap = new WeakMap<HTMLElement, number>();
  const root = document.createElement('span');
  root.className = 'expr-root';

  leaves.forEach((val, i) => {
    const sym = document.createElement('span');
    sym.className = 'expr-symbol';
    sym.appendChild(createSymbolEl(val));
    root.appendChild(sym);

    if (i < leaves.length - 1) {
      const opEl = document.createElement('span');
      opEl.className = 'expr-op flat-redex';
      opEl.tabIndex = 0;
      opEl.append(' ', createOperatorEl(), ' ');
      opIndexMap.set(opEl, i);
      root.appendChild(opEl);
    }
  });

  return { element: root, opIndexMap };
}

For a tree with `n` leaves this creates `O(n)` intermediate arrays and does `O(n²)` copying via spread in the worst case (right-skewed tree). The standard fix is an accumulator:

```typescript

  if (expr.kind === 'symbol') { out.push(expr.value); return out; }
  flattenLeaves(expr.left, out);
  flattenLeaves(expr.right, out);
  return out;
}
```

With `n = 5` leaves this does not matter in practice, but it is a style issue worth noting.

### 8. `.then(() => {})` calls are unnecessary

In `flyBetween`:


`Animation.finished` already returns a `Promise<Animation>`. The `.then(() => {})` calls discard the `Animation` value to produce `Promise<void>`, which is then put into `Promise<void>[]`. return Promise.all([


The three `.then(() => {})` calls each allocate an extra microtask and promise object for no reason other than type-level `void` coercion. The outer `Promise.all(...).then(() => {})` is similarly just a type-cast; a single `void` coercion at the return site is sufficient.

## src/symbols.ts

### 9. `SYMBOL_GLYPH_IDS` and `OPERATOR_GLYPH_IDS` are unnecessary arrays


These allocate 1000-element and 100-element arrays at module load time. They are used in exactly two ways:

2. `OPERATOR_GLYPH_IDS` is only used to check `.length > 0` and to index with `Math.floor(Math.random() * OPERATOR_GLYPH_IDS.length)`.

```typescript
const SYMBOL_GLYPH_COUNT = 1000;
const OPERATOR_GLYPH_COUNT = 100;
```

Then `randomizeGlyphs` builds the Fisher-Yates pool on the fly with a size-bounded array (`Array.from({ length: SYMBOL_GLYPH_COUNT }, (_, i) => i)` — or inline), and the operator pick becomes `Math.floor(Math.random() * OPERATOR_GLYPH_COUNT)`. This eliminates the two module-level allocations entirely.

The current `OPERATOR_GLYPH_IDS.length > 0` guard is always true (the array always has 100 elements), so that branch is dead code that could be removed if the count constant approach is adopted.

## src/minizinc/generate.ts

### 10. Random objective weights — correctness and efficiency

```typescript
const weights = Array.from({ length: symbolCount * symbolCount }, () =>
  Math.floor(Math.random() * 201) - 100
);
const objectiveTerms = weights.map((w, idx) => {
  const r = Math.floor(idx / symbolCount) + 1;
  const c = (idx % symbolCount) + 1;
  return `${String(w)} * op[${String(r)},${String(c)}]`;
});
const solve = `solve maximize ${objectiveTerms.join(" + ")};`;
```

**Correctness:** The approach is sound. Adding a random linear objective over the table cells biases the solver toward different feasible solutions on each call without changing the feasible set. HiGHS (a LP/MIP solver) will generally find the solution that maximises the objective subject to constraints, so different random weights should indeed produce different tables. This is a well-known trick for solution diversity.

**Minor issue — zero weights generate needless terms.** With weights in `[-100, 100]` there is a `1/201` chance any weight is zero, producing terms like `"0 * op[2,3]"` in the objective. These are harmless but slightly wasteful. Filtering them out would be a trivial cleanup.

**Minor issue — intermediate `weights` array.** `weights` is created and then immediately mapped again to produce `objectiveTerms`. The two steps could be combined into a single `Array.from` with the index-to-`(r,c)` mapping inline, avoiding the intermediate array. This is a style/efficiency micro-concern at a max size of 16 elements (4×4), so it has no practical impact.

**Structural note:** `symbolCount` is validated to be in `[2, 4]` before `generateMzn` is called, so `symbolCount * symbolCount` is at most 16. The approach is completely fine at this scale.
