### 3. Key Data Types

#### 3.1 Operation Table (`src/table/types.ts`)

```typescript
/**
 * A binary operation on symbols 0..n-1.
 * op[i][j] gives the result of applying the operation to symbol i and symbol j.
 * Stored 0-indexed internally (converted from the solver's 1-indexed output).
 */
export interface OpTable {
  readonly size: number;           // always 3 for now
  readonly cells: readonly (readonly number[])[];  // size x size, 0-indexed
}

/** Look up the result of a op b. */
export function lookup(table: OpTable, a: number, b: number): number {
  return table.cells[a][b];
}

/** Convert solver result (1-indexed) to OpTable (0-indexed). */
export function fromSolverResult(op: number[][]): OpTable {
  return {
    size: op.length,
    cells: op.map(row => row.map(v => v - 1)),
  };
}

/** Generate a purely random operation table (no constraints). */
export function randomTable(size: number): OpTable {
  return {
    size,
    cells: Array.from({ length: size }, () =>
      Array.from({ length: size }, () => Math.floor(Math.random() * size))
    ),
  };
}
```

The 1-to-0 index conversion is important because the solver schema uses `z.number().int().min(1)` -- values are in range `1..n`. The UI should work with 0-indexed symbols internally and map to display names at render time.

#### 3.2 Expression Tree (`src/expr/tree.ts`)

```typescript
/** A symbol leaf, identified by its 0-based index. */
export interface SymbolNode {
  readonly kind: 'symbol';
  readonly value: number;        // 0, 1, or 2
}

/** A binary operation node. */
export interface OpNode {
  readonly kind: 'op';
  readonly left: Expr;
  readonly right: Expr;
}

export type Expr = SymbolNode | OpNode;

/** Create a symbol node. */
export function sym(value: number): SymbolNode {
  return { kind: 'symbol', value };
}

/** Create an operation node. */
export function op(left: Expr, right: Expr): OpNode {
  return { kind: 'op', left, right };
}

/**
 * A redex is an OpNode whose left and right children are both SymbolNodes.
 * This is the only kind of sub-expression the user can click to reduce.
 */
export function isRedex(node: Expr): node is OpNode & { left: SymbolNode; right: SymbolNode } {
  return node.kind === 'op' && node.left.kind === 'symbol' && node.right.kind === 'symbol';
}
```

Key design decision: a "redex" (reducible expression) is defined as an `OpNode` where **both children are `SymbolNode`s**. This means the user must reduce innermost operations first, naturally enforcing a step-by-step reduction from the leaves inward. This is similar to how leftmost-innermost reduction works in lambda calculus.

#### 3.3 Reduction

```typescript
/** 
 * Reduce a specific redex within an expression tree.
 * Uses reference identity to find the target node.
 * Returns a new tree with the redex replaced by its result symbol.
 */
export function reduce(expr: Expr, target: OpNode, table: OpTable): Expr {
  if (expr === target && isRedex(expr)) {
    const result = lookup(table, expr.left.value, expr.right.value);
    return sym(result);
  }
  if (expr.kind === 'op') {
    return op(reduce(expr.left, target, table), reduce(expr.right, target, table));
  }
  return expr;
}

/** Collect all redexes in the tree (for highlighting clickable regions). */
export function findRedexes(expr: Expr): OpNode[] {
  if (expr.kind === 'symbol') return [];
  const result: OpNode[] = [];
  if (isRedex(expr)) {
    result.push(expr);
  } else {
    result.push(...findRedexes(expr.left));
    result.push(...findRedexes(expr.right));
  }
  return result;
}

/** Is the expression fully reduced (a single symbol)? */
export function isFullyReduced(expr: Expr): expr is SymbolNode {
  return expr.kind === 'symbol';
}
```

The `reduce` function uses **reference identity** (`===`) to find the target node. This works because expression trees are immutable -- each call to `reduce` returns a fresh tree, and the UI holds a reference to the specific `OpNode` the user clicked.

#### 3.4 Random Expression Generation (`src/expr/tree.ts`)

```typescript
/**
 * Generate a random expression tree with a target number of leaves.
 * A tree with `n` leaves has `n-1` internal (op) nodes, giving `n-1` reduction steps.
 * 
 * Strategy: start with `leafCount` symbol leaves, then repeatedly
 * pick two adjacent items and combine them into an OpNode.
 */
export function randomExpr(symbolCount: number, leafCount: number): Expr {
  // Start with random symbol leaves
  const leaves: Expr[] = Array.from(
    { length: leafCount },
    () => sym(Math.floor(Math.random() * symbolCount))
  );
  
  // Randomly build tree structure by combining adjacent pairs
  while (leaves.length > 1) {
    const i = Math.floor(Math.random() * (leaves.length - 1));
    const combined = op(leaves[i], leaves[i + 1]);
    leaves.splice(i, 2, combined);
  }
  
  return leaves[0];
}
```

A `leafCount` of 4-6 gives good expressions: 3-5 reduction steps, complex enough to be interesting without being overwhelming. The random tree construction by combining adjacent items produces varied tree shapes (not just left-skewed or right-skewed).

#### 4.2 Expression Rendering (`src/expr/render.ts`)

Each node in the expression tree maps to a DOM element:

- **SymbolNode** -> `<span class="symbol" data-value="0">X</span>`
- **OpNode (redex)** -> `<span class="redex" tabindex="0" role="button">(</span>` + left + `<span class="op-sym">&#9675;</span>` + right + `<span>)</span>`, all wrapped in a `<span class="redex-group">` container
- **OpNode (non-redex)** -> same structure but class `expr-group` instead of `redex-group`, no tabindex/role

The critical insight: **each rendered redex-group holds a reference to its `OpNode`** via a `WeakMap<HTMLElement, OpNode>` or a closure in the click handler. When the user clicks a redex-group, we know exactly which `OpNode` in the tree to reduce.

```typescript
/** Maps DOM elements to their corresponding OpNodes for click handling. */
const nodeMap = new WeakMap<HTMLElement, OpNode>();

/**
 * Render an Expr tree into a DOM fragment.
 * Returns the root element.
 */
export function renderExpr(
  expr: Expr,
  symbols: readonly string[],
  opSymbol: string
): HTMLElement {
  if (expr.kind === 'symbol') {
    const span = document.createElement('span');
    span.className = 'symbol';
    span.dataset.value = String(expr.value);
    span.textContent = symbols[expr.value];
    return span;
  }

  const container = document.createElement('span');
  const redex = isRedex(expr);
  container.className = redex ? 'redex-group' : 'expr-group';

  if (redex) {
    container.tabIndex = 0;
    container.setAttribute('role', 'button');
    container.setAttribute('aria-label', 
      `Reduce ${symbols[(expr.left as SymbolNode).value]} ${opSymbol} ${symbols[(expr.right as SymbolNode).value]}`
    );
    nodeMap.set(container, expr);
  }

  // Opening paren
  const open = document.createElement('span');
  open.className = 'paren';
  open.textContent = '(';
  container.appendChild(open);

  // Left child
  container.appendChild(renderExpr(expr.left, symbols, opSymbol));

  // Operator
  const opSpan = document.createElement('span');
  opSpan.className = 'op-sym';
  opSpan.textContent = ` ${opSymbol} `;
```

```typescript
export function renderTable(
  table: OpTable,
  symbols: readonly string[],
  opSymbol: string
): HTMLTableElement {
  const el = document.createElement('table');
  el.className = 'op-table';

  // Header row: [op] [sym0] [sym1] [sym2]

    const row = tbody.insertRow();
    const rowHeader = document.createElement('th');

    row.appendChild(rowHeader);

      cell.textContent = symbols[table.cells[i][j]];
      cell.dataset.row = String(i);
      cell.dataset.col = String(j);

    }
  }

  return el;
}
```

### 5. Animation Strategy (`src/expr/animate.ts`)

The reduction animation has four phases, executed sequentially:

**Phase 1: Highlight the table cell (200ms)**
- The cell at `[left.value][right.value]` in the operation table gets class `cell-flash`.
- CSS: scale up slightly, bright background glow, then settle.

**Phase 2: Animate the result symbol flying from the table to the expression (400ms)**
- Create a **floating clone** of the result symbol, absolutely positioned at the table cell's coordinates.
- Use the Web Animations API (`element.animate()`) to translate it from the table cell position to the redex-group position in the equation.
- The clone uses `position: fixed` so it can travel across layout boundaries.

**Phase 3: Collapse the redex (300ms)**
- The redex-group fades out its parentheses, operator, and operand symbols (opacity -> 0, width -> 0).
- Simultaneously, the result symbol fades in at the same position.
- Use CSS `transition` on `max-width` and `opacity` for smooth collapsing, or Web Animations API for more precise control.

**Phase 4: Re-space (200ms)**
- The parent containers smoothly adjust their widths as the collapsed content is removed.
- This happens naturally via CSS transitions on `gap` / flex layout if the elements are in a flex container.

Implementation approach:

```typescript
export interface AnimationContext {
  tableEl: HTMLTableElement;
  equationEl: HTMLElement;
}

export async function animateReduction(
  ctx: AnimationContext,
  redexEl: HTMLElement,
  left: number,
  right: number,
  result: number,
  symbols: readonly string[]
): Promise<void> {
  // Phase 1: Flash the table cell
  const cell = ctx.tableEl.querySelector(`#cell-${left}-${right}`) as HTMLElement;
  cell.classList.add('cell-flash');
  await delay(200);

  // Phase 2: Fly the result symbol
  const cellRect = cell.getBoundingClientRect();
  const redexRect = redexEl.getBoundingClientRect();

  const floater = document.createElement('span');
  floater.className = 'symbol floater';
  floater.textContent = symbols[result];
  floater.style.position = 'fixed';
  floater.style.left = `${cellRect.left + cellRect.width / 2}px`;
  floater.style.top = `${cellRect.top + cellRect.height / 2}px`;
  floater.style.transform = 'translate(-50%, -50%)';
  floater.style.zIndex = '1000';
  document.body.appendChild(floater);

  const targetX = redexRect.left + redexRect.width / 2;
  const targetY = redexRect.top + redexRect.height / 2;

  await floater.animate([
    { left: `${cellRect.left + cellRect.width / 2}px`, top: `${cellRect.top + cellRect.height / 2}px`, opacity: 1 },
    { left: `${targetX}px`, top: `${targetY}px`, opacity: 1 },
  ], { duration: 400, easing: 'ease-in-out', fill: 'forwards' }).finished;

  floater.remove();

  // Phase 3: Collapse the redex, replace with result symbol
  redexEl.classList.add('collapsing');
  
  // Insert the result symbol adjacent to the redex
  const resultSpan = document.createElement('span');
  resultSpan.className = 'symbol result-appear';
  resultSpan.dataset.value = String(result);
  resultSpan.textContent = symbols[result];
  redexEl.parentNode!.insertBefore(resultSpan, redexEl);

  // Animate collapse
  await redexEl.animate([
    { maxWidth: `${redexEl.offsetWidth}px`, opacity: 1 },
    { maxWidth: '0px', opacity: 0 },
  ], { duration: 300, easing: 'ease-in-out', fill: 'forwards' }).finished;

  redexEl.remove();
  cell.classList.remove('cell-flash');

  // Phase 4: result-appear animates in via CSS
  resultSpan.classList.add('settled');
  await delay(200);
}

function delay(ms: number): Promise<void> {
  return new Promise(resolve => setTimeout(resolve, ms));
}
```

There is an important subtlety with **Phase 3**: rather than re-rendering the entire expression tree after each reduction, we perform a **surgical DOM update** -- removing the redex-group element and inserting a result symbol in its place. The expression tree in memory is updated separately (via the pure `reduce` function), and the next set of redex click handlers is attached after the animation completes. This avoids a jarring full re-render.

However, the simpler alternative is to re-render the entire expression after each animation. Since expressions are small (at most ~10 nodes), the cost is trivial, and it avoids bugs from DOM/tree state getting out of sync. The choice between surgical and full re-render should be made during implementation. I recommend **full re-render after animation completes** for correctness, with a brief crossfade to smooth the transition.

```typescript
function renderEquation(container: HTMLElement): void {
  container.innerHTML = '';
  const exprEl = renderExpr(state.expr, SYMBOLS, OP_SYMBOL);

function attachRedexHandlers(container: HTMLElement): void {
  const redexEls = container.querySelectorAll('.redex-group');
  for (const el of redexEls) {
    el.addEventListener('click', handleRedexClick);
    el.addEventListener('keydown', (e) => {
      if ((e as KeyboardEvent).key === 'Enter' || (e as KeyboardEvent).key === ' ') {
        e.preventDefault();
        handleRedexClick.call(el, e as Event);
      }
    });

async function handleRedexClick(this: Element, _e: Event): Promise<void> {
  if (state.animating) return;
  state.animating = true;

  const el = this as HTMLElement;
  const node = getNode(el);
  if (!node || !isRedex(node)) { state.animating = false; return; }

  const left = node.left.value;
  const right = node.right.value;
  const result = state.table.cells[left][right];

  const tableEl = document.querySelector('.op-table') as HTMLTableElement;
  const equationEl = document.querySelector('.equation') as HTMLElement;
  const ctx: AnimationContext = { tableEl, equationEl };

  await animateReduction(ctx, el, left, right, result, SYMBOLS);

  // Update the tree
  state.expr = reduce(state.expr, node, state.table);

  // Re-render the equation from the updated tree
  renderEquation(equationEl);

  state.animating = false;

  if (isFullyReduced(state.expr)) {
    celebrateCompletion(equationEl);
  }
}

  // Brief "done" animation -- glow/pulse on the final symbol
  const sym = container.querySelector('.symbol');
  if (sym) sym.classList.add('final-result');
}
```

### 10. Key Design Decisions and Trade-offs

**Immutable expression trees with reference identity for targeting**: The `reduce` function finds its target by `===` comparison. This is elegant and avoids needing path-based addressing or unique IDs on nodes. The downside is that every reduction creates a completely new tree, but with at most ~10 nodes this is negligible.

**Full re-render vs. surgical DOM update after animation**: I recommend full re-render for correctness. The animation does its visual work on the existing DOM, then after the animation finishes, we update the tree and re-render. There is a brief moment where the DOM switches from the "animated state" to the "clean re-render" -- if this is jarring, we can add a short crossfade. But because the animation's final state and the re-render's initial state should look identical (both show the same reduced expression), the transition should be seamless.

**Random tables vs. solver**: For the initial implementation, `randomTable` is simpler and faster (no WASM initialization). We can add a "Generate with constraints" feature later that uses the solver. The `fromSolverResult` converter is already defined for when we want to integrate.

**0-indexed internal, display names at the boundary**: The solver uses 1-indexed values. We convert to 0-indexed immediately in `fromSolverResult`. All internal logic uses 0-indexed. Display names are applied only at render time. This prevents off-by-one bugs.

**CSS colors per symbol via data attributes**: Rather than setting inline styles, we use `data-value` attributes and CSS rules. This keeps style logic in CSS and makes it easy to change the color scheme later.

**Web Animations API over CSS keyframe animations**: The flying-symbol animation requires knowing runtime coordinates (table cell position, redex position). The Web Animations API lets us specify these dynamically. CSS `@keyframes` cannot accept runtime values. The cell-flash and final-celebration effects can use CSS animations since they do not depend on layout.

# Interactive Redex Reduction UI

## Context

The newmath project has a working MiniZinc constraint solver but no real frontend yet — just the Vite scaffold (counter demo). This plan builds the first user-facing feature: an interactive UI where users see a 3×3 operation table and reduce algebraic expressions step-by-step by clicking on innermost sub-expressions (redexes).

## New file structure

```
src/
  symbols.ts              — symbol display names, colors, operation symbol
  expr/
    CLAUDE.md
    tree.ts               — Expr tree type, redex detection, reduction, random generation
    render.ts             — Expr → DOM with clickable redex targets (WeakMap tracking)
    animate.ts            — 4-phase reduction animation (flash → fly → collapse → re-space)
  table/
    CLAUDE.md
    types.ts              — OpTable type, lookup, randomTable, fromSolverResult
    render.ts             — OpTable → DOM <table> element
```

Files modified: `src/main.ts`, `src/style.css`, `index.html`, `src/CLAUDE.md`
Files deleted: `src/counter.ts`, `src/typescript.svg`, `public/vite.svg`

## Key data types

### Expression tree (`src/expr/tree.ts`)

```typescript
type Expr = SymbolNode | OpNode
interface SymbolNode { kind: 'symbol'; value: number }  // 0-indexed
interface OpNode { kind: 'op'; left: Expr; right: Expr }
```

- A **redex** = OpNode where both children are SymbolNodes (innermost binary op)
- `reduce(expr, targetNode, table)` finds target by reference identity (`===`), returns new tree with redex replaced by result symbol
- `randomExpr(symbolCount, leafCount)` builds a random tree by combining adjacent leaves — 4–6 leaves gives 3–5 reduction steps

### Operation table (`src/table/types.ts`)

```typescript
interface OpTable { size: number; cells: number[][] }  // 0-indexed
```

- `randomTable(3)` for initial version (no solver needed)
- `fromSolverResult(op)` converts solver's 1-indexed output to 0-indexed

## Rendering approach

- **Expression**: Each tree node → a `<span>`. Redex groups get `class="redex-group"` + `tabindex="0"` + click handler. A `WeakMap<HTMLElement, OpNode>` maps DOM elements to tree nodes.
- **Table**: Standard `<table>` with `id="cell-{row}-{col}"` on each td for animation targeting. Each cell gets `data-value` for CSS color rules.
- **After each reduction**: Full re-render of the expression from the updated tree (simple, correct, negligible cost for ~10 nodes).

## Animation sequence (`src/expr/animate.ts`)

When user clicks a redex `(A ○ B)`:

1. **Flash** (200ms): Cell `[A][B]` in table gets `.cell-flash` — scale up + glow
2. **Fly** (400ms): Clone result symbol at cell position, Web Animations API translates it to redex position (`position: fixed`)
3. **Collapse** (300ms): Redex-group animates `max-width → 0, opacity → 0`. Result symbol inserted adjacent, fades in.
4. **Re-space** (200ms): Flex layout naturally adjusts. Remove old DOM, update tree, full re-render.

Lock flag (`state.animating`) prevents clicks during animation.

## Visual design

- **Background**: true black `#000`
- **Font**: Cormorant Garamond (Google Fonts) — elegant serif with math feel
- **Symbol colors**: X = `#ff6b6b` (red), Y = `#4ecdc4` (teal), Z = `#ffe66d` (gold)
- **Layout**: CSS Grid — table left, equation right on wide screens; stacked on mobile
- **Redex hover**: subtle purple glow indicating clickability
- **Completion**: final symbol pulses with a glow animation
