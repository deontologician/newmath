% Find an associative binary operation on 3 symbols
% where exactly one symbol is a right-zero (but not a left-zero).
%
% Right-zero z:  for all a, a * z = z
% "Only" means:  z is NOT also a left-zero  (exists a : z * a != z)
%                and no other element is a right-zero.

int: n = 3;
set of int: S = 1..n;

% Operation table: op[a,b] represents a * b
array[S, S] of var S: op;

% --- Associativity ---
% (a * b) * c  =  a * (b * c)   for all a, b, c
constraint forall(a in S, b in S, c in S)(
    op[op[a, b], c] = op[a, op[b, c]]
);

% --- Exactly one right-zero element ---
var S: rz;

% rz is a right-zero: for all a, a * rz = rz
constraint forall(a in S)(
    op[a, rz] = rz
);

% rz is NOT a left-zero: there exists some a where rz * a != rz
constraint exists(a in S)(
    op[rz, a] != rz
);

% No other element is a right-zero
constraint forall(z in S where z != rz)(
    exists(a in S)( op[a, z] != z )
);

solve satisfy;

output [
    "Symbols: {1, 2, 3}\n",
    "Right-zero element: \(rz)\n\n",
    "Operation table (row * col):\n",
    "  * | " ++ join("  ", [show(j) | j in S]) ++ "\n",
    " ---+--------\n"
] ++ [
    "  \(i) | " ++ join("  ", [show(op[i, j]) | j in S]) ++ "\n"
    | i in S
] ++ [
    "\nAll pairs:\n"
] ++ [
    "  \(i) * \(j) = \(op[i, j])\n"
    | i in S, j in S
];

### 2. Types (`src/minizinc/types.ts`)

The key design question is: should a constraint be a simple string, or an object with metadata? The answer is an object, but a minimal one. The reason is that "existential" constraints (identity, zero) need to introduce a MiniZinc variable via `exists(e in S)(...)`, while universal constraints (associativity, commutativity) are plain `forall` blocks. Both produce a single `constraint ...;` statement in MiniZinc, but having metadata (a name for the comment, a description for documentation) is useful and costs nothing.

```typescript
  readonly description: string;
  readonly mzn: () => string;
```

**Section 1: Preamble**

**Section 2: Constraints**

Each constraint's `mzn()` output, separated by a blank line:


**Section 3: Solve directive**

```minizinc
solve satisfy;
```

**Section 4: Output section**

This is always the same structure, matching the hand-written file. It adapts to `n` only through the MiniZinc `S = 1..n` set, so the output template is static MiniZinc:

```minizinc
output [
    "Operation table (row * col):\n",
    "  * | " ++ join("  ", [show(j) | j in S]) ++ "\n",
    " ---+" ++ join("", ["---" | _ in S]) ++ "\n"
] ++ [
    "  \(i) | " ++ join("  ", [show(op[i, j]) | j in S]) ++ "\n"
    | i in S
] ++ [
    "\nAll pairs:\n"
] ++ [
    "  \(i) * \(j) = \(op[i, j])\n"
    | i in S, j in S
];
```

Note the separator line `" ---+" ++ join("", ["---" | _ in S])` -- this makes the table divider scale with `n` instead of being hardcoded to `"--------"`. The original file hardcoded it for `n=3`; generating it dynamically is better.

**Full function implementation sketch:**

```typescript
import type { GeneratorConfig } from "./types.ts";

export function generateMzn(config: GeneratorConfig): string {
  const { symbolCount, constraints } = config;

  if (symbolCount < 2 || symbolCount > 4) {
    throw new RangeError(`symbolCount must be 2-4, got ${symbolCount}`);
  }

  const header = [
    `% Generated by newmath`,
    `% Constraints: ${constraints.map((c) => c.name).join(", ")}`,
    ``,
    `int: n = ${symbolCount};`,
    `set of int: S = 1..n;`,
    ``,
    `% Operation table: op[a,b] represents a * b`,
    `array[S, S] of var S: op;`,
  ].join("\n");

  const body = constraints.map((c) => c.mzn()).join("\n\n");

  const solve = `solve satisfy;`;

  const output = `output [
    "Operation table (row * col):\\n",
    "  * | " ++ join("  ", [show(j) | j in S]) ++ "\\n",
    " ---+" ++ join("", ["---" | _ in S]) ++ "\\n"
] ++ [
    "  \\(i) | " ++ join("  ", [show(op[i, j]) | j in S]) ++ "\\n"
    | i in S
] ++ [
    "\\nAll pairs:\\n"
] ++ [
    "  \\(i) * \\(j) = \\(op[i, j])\\n"
    | i in S, j in S
];`;

  return [header, body, solve, output].join("\n\n") + "\n";
}
```

An important detail: the MiniZinc string interpolation syntax uses `\(expr)` and the literal `\n`. Inside a TypeScript template literal, backslashes must be escaped. The `output` variable above uses `\\n` and `\\(i)` so that the generated `.mzn` file contains literal `\n` and `\(i)`. This is a common source of bugs and needs care during implementation.

### 8. Design Decisions and Trade-offs

**Why `mzn` is `() => string` not just `string`.** Keeps the option open for parameterized constraints (e.g., `idempotentFor(k: number)` returning a `Constraint`). Zero cost now, saves a refactor later.

**Why no separate "declarations" and "body" fields on Constraint.** MiniZinc's `exists(v in S)(...)` introduces quantified variables inline. There is no need to declare them separately. All six initial constraints fit the single-block pattern. If a future constraint truly needs a top-level `var` declaration (unlikely for existential algebraic properties), we can add an optional `declarations` field then.

**Why validate symbolCount in the generator, not in a separate config validator.** Fail-fast at the single call site. There is only one entry point (`generateMzn`), so validation there catches all errors.

**Why the output section is hardcoded in the generator rather than configurable.** The output format is a property of the problem family (binary operation tables), not of individual constraints. Every problem in this family wants the same table display. If the project later needs different output formats, the output section can be extracted into a strategy, but that would be premature now.

**Why the table separator uses `join("", ["---" | _ in S])` instead of a hardcoded string.** The original file hardcodes `"--------"` which only works for `n=3`. Since `n` is now configurable (2-4), the separator must scale. Using a MiniZinc comprehension is the cleanest way.

## New Dependencies

## File Structure

```
src/minizinc/
  CLAUDE.md          — directory docs
  schema.ts          — Zod schemas (config in, results out)
  constraints.ts     — constraint library (each returns MiniZinc text)
```

## Detailed Design

### 1. `schema.ts` — Zod Schemas

```typescript
// --- Input: what the UI sends ---
const ConstraintName = z.enum([
  "associativity",
  "commutativity",
  "leftIdentity",
  "rightIdentity",
  "leftZero",
  "rightZero",
]);

const SolverConfig = z.object({
  symbolCount: z.number().int().min(2).max(4),
  constraints: z.array(ConstraintName).min(1),
});

// --- Output: what the solver returns ---
// MiniZinc JSON mode returns op as a nested array: [[1,2],[2,1]]
const SolverResult = z.object({
  op: z.array(z.array(z.number().int().positive())),
});
```

Types are inferred with `z.infer<typeof SolverConfig>` / `z.infer<typeof SolverResult>` — no duplicate type definitions.

### 2. `constraints.ts` — Constraint Library

A record mapping each `ConstraintName` to a function returning MiniZinc text. Each constraint is a self-contained `constraint ...;` block (existential constraints like identity/zero use MiniZinc's `exists(e in S)(...)` — no separate variable declarations needed).

```
constraintRegistry: Record<ConstraintName, { description: string; mzn: string }>
```

| Name | MiniZinc |
|---|---|
| `associativity` | `forall(a,b,c in S)( op[op[a,b],c] = op[a,op[b,c]] )` |
| `commutativity` | `forall(a,b in S)( op[a,b] = op[b,a] )` |
| `leftIdentity` | `exists(e in S)( forall(a in S)( op[e,a] = a ))` |
| `rightIdentity` | `exists(e in S)( forall(a in S)( op[a,e] = a ))` |
| `leftZero` | `exists(z in S)( forall(a in S)( op[z,a] = z ))` |
| `rightZero` | `exists(z in S)( forall(a in S)( op[a,z] = z ))` |

### 3. `generate.ts` — MiniZinc Generator

Single function: `generateMzn(config: SolverConfig): string`

Composes four sections:
1. **Preamble**: `int: n = <symbolCount>; set of int: S = 1..n; array[S,S] of var S: op;`
2. **Constraints**: looked up from `constraintRegistry` by name
3. **Solve**: `solve satisfy;`
4. **Output**: human-readable table (used only for CLI debugging; WASM path uses JSON mode)

Throws `RangeError` if symbolCount is out of range. Config is already Zod-validated before reaching here, but defense in depth.

### 4. `solve.ts` — WASM Solver

```typescript
async function solve(config: SolverConfig): Promise<SolverResult>
```

Steps:
1. Validate `config` with `SolverConfig.parse()`
2. Call `generateMzn(config)` to get the .mzn string
3. Create a `MiniZinc.Model`, add the .mzn string via `model.addFile()`
4. Call `model.solve()` (uses `--output-mode json` by default)
5. Validate `result.solution.output.json` with `SolverResult.parse()`
6. Return the typed result

For all-solutions mode (like the `-a` flag used earlier), use `{ options: { 'all-solutions': true } }` and collect solutions via the `'solution'` event.

### 5. `index.ts` — Barrel

Re-exports schemas, types, constraint registry, `generateMzn`, and `solve`.

### 6. WASM File Serving

The `minizinc` package requires 3 files to be web-accessible: `minizinc-worker.js`, `minizinc.wasm`, `minizinc.data`. Copy these from `node_modules/minizinc/dist/` into `public/wasm/` and configure `MiniZinc.init()` with explicit URLs. Add a `postinstall` script in package.json to automate the copy.
